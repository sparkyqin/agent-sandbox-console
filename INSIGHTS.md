# Agent 沙箱控制台 · 调研洞察报告

> 这份报告不是设计规格（那是 [DESIGN.md](./DESIGN.md) 的职责），而是调研过程中**提炼出的洞察与避坑点**——那些容易被自研方案忽视、或做错、值得铭记的判断。每条都给出"常见误区 → 业界实践 → 为什么"。入门请先看 [README.md](./README.md)。

调研对象：E2B、Daytona、Modal、Runloop、CodeSandbox SDK、Morph、Fly Machines 等沙箱即服务平台，以及 Firecracker、gVisor、Kata、Kubernetes（Pod/Probe/Resource/NetworkPolicy/RBAC/CSI/CRIU）、KEDA、Knative 等底层实践。

---

## 洞察 1：供给 ≠ 管理 —— 最大且最普遍的认知错位

**常见误区**：把"沙箱控制台"做成一个创建向导。配好参数 → 点创建 → 弹"成功" → 结束。沙箱去哪了？不知道。

**业界实践**：E2B/Daytona/Modal/Runloop 的控制面核心都是 `list` + `get` + 生命周期操作 + 可观测性。创建只是入口，**运行态的管理才是控制台存在的理由**。实例列表（状态/资源/就绪/重启/花费）+ 实例详情（日志/终端/监控/事件/快照）是一等公民。

**为什么**：沙箱是**有状态的长生命周期对象**，不是"提交即结束"的无状态请求。Agent 会跑几小时甚至几天，期间要观测、要干预、要回收。一个只管创建的界面，等于把沙箱变成了"发射后不管"的火箭。

**避坑**：如果资源只够做一个页面，做**实例列表**，不要做创建页。创建可以靠 CLI/SDK 兜底，但"我的沙箱在哪、什么状态、花了多少钱"没有替代品。

## 洞察 2：快照不是一种东西，是两种

**常见误区**：把"快照"当成单一概念，UI 上一个"打快照"按钮了事。

**业界实践**：一个完整的有状态沙箱快照需要**两类机制组合**，业界明确二分：

| 类型 | 捕获 | 机制 | 恢复速度 |
|---|---|---|---|
| **Warm（内存快照）** | 进程树 + 全部内存页 + FD + 打开文件 + socket 状态 | CRIU（KEP-2008）/ Firecracker 内存快照 | 秒级（~1.5s） |
| **Cold（磁盘快照）** | 仅存储卷 | CSI VolumeSnapshot（GA） | 冷启（~6s） |

**为什么**：Agent 经常需要"保留当前运行态，从这一点分叉探索多个方案"。这只能靠 Warm 快照——它把内存里的进程、变量、打开的连接全部冻住，恢复后**接着跑**。Cold 快照只能恢复文件系统，进程得重启。这两者语义完全不同，混为一谈会让用户在错误场景拿到错误结果。

**避坑**：
- UI 要显式区分 "Warm 快照" vs "Cold 快照"
- 对应地，派生操作要分 **Warm Fork**（保内存、秒级、可 1-to-many）vs **Cold Clone**（仅磁盘、冷启）
- 实例详情页要有 **Fork 血缘树**——这是 Agent 试错（"分支化探索"）场景最被低估的能力，E2B/Runloop/Daytona/Morph/CodeSandbox 都有，多数自研控制台漏掉

**技术澄清**（调研中发现的常见误传）：容器检查点的正确 KEP 是 **KEP-2008**（"Forensic Container Checkpointing"，Beta，v1.30 默认开）；Pod 级检查点是 **KEP-5823**（Alpha，v1.37）。网上常被误引的 "KEP-3157" 是无关的 `watch-list` KEP，别照抄。

## 洞察 3：idle 判定 —— Colab 是反面教材

**常见误区**：照搬 Google Colab 的 idle 模型，以为"用户不操作了就挂起"是通用最佳实践。

**业界实践（反面）**：Colab 用**浏览器 tab 交互**判定 idle，所有档位共享 ~90min 超时，Pro 不延长，**而且 GPU 跑满也不算活动**。这对交互式 notebook 够用，但对 Agent 是灾难。

**业界实践（正面）**：Knative 用**两窗口模型**——`grace period`（缩到 0 后保留时长，默认 30s）+ `stable window`（判定稳定空闲的窗口，默认 60s），基于**真实并发/请求活动**判定。

**为什么**：Agent 是**无人值守长跑**的。它可能在 GPU 上跑 30 分钟训练，期间没有任何"用户交互"，但它在满负荷工作。用 UI 交互判 idle 会**误杀正在计算的 agent**——这是最伤体验的 bug。

**避坑**：
- idle 判定必须基于**真实资源活动**：CPU 利用率、网络流量、并发请求数
- 用两窗口模型（grace + stable），不要单一 timeout
- 给用户暴露 idle 超时为**显式可配字段**（Daytona 的 `autoStop`、Fly 的 `auto_stop`），而非全局硬编码

## 洞察 4：`auto_stop` + `auto_start` + `restart_policy` 三件套必须显式

**常见误区**：创建时不暴露任何生命周期治理参数，靠"用户记得手动关"或"全局默认超时"。

**业界实践**：Daytona、Fly 把三件事都做成创建时的**显式字段**：
- `autoStop`：空闲多久自动挂起（保留磁盘，省钱）
- `autoStart`：访问时是否自动唤醒（低延迟）
- `restart_policy`：崩溃是否自动重启（健壮性）

**为什么**：Agent 会话常常是**长生命周期 + 间歇活动**的——工作 10 分钟，闲 2 小时，再工作。没有 auto-stop，闲置 2 小时也在烧钱；没有 auto-start，唤醒要等冷启动；没有 restart-policy，agent 进程崩了就永久挂了。这三件套是"长会话 Agent"省心省钱的标准配置。

**避坑**：别只给一个 `timeout`（超时销毁）。销毁是**最激进**的回收——状态全丢。应该给一档阶梯：`idle → 挂起(保留磁盘) → 销毁(全丢)`，让用户选挂在哪一档。

## 洞察 5：AI Agent 沙箱的安全风险比普通容器更高，不是更低

**常见误区**：觉得"沙箱本来就是隔离的，安全差不多"。或者反过来，只做隔离不管出口。

**业界实践**：Agent 沙箱有两个普通容器没有的放大因素：
1. **执行不可信代码**：Agent 会动态生成并运行代码，可能含漏洞或恶意逻辑
2. **自主发起网络请求**：Agent 会自己决定访问哪些 URL，可能被提示注入诱导外泄数据

所以控制台必须显式可配：
- **egress 出口白名单**（防数据外泄）—— 这是 Agent 场景的**安全关键项**，比入口控制更重要
- **Secret 与明文 env 分离**（加密存储 → 注入，不在日志回显）
- **max_lifetime 强制销毁**（防失控成本 + 降低逃逸窗口）
- **资源 limits**（防单实例耗尽宿主机）

**为什么**：一个被提示注入的 agent，可能把宿主上能拿到的所有数据 POST 到攻击者服务器。没有 egress 白名单，隔离再强也挡不住"合法进程发合法请求"。这是 Agent 沙箱区别于普通 dev 容器的核心安全差异。

**避坑**：
- 默认 egress 策略应是"白名单"而非"完全开放"
- `docker-cli` 这类 DooD（挂载宿主 Docker）工具要标注"高权限"、默认禁用
- 明文 env 存密钥是常见反模式，必须改 Secret 机制

## 洞察 6：端口暴露的"自动公开 URL"是刚需，不是锦上添花

**常见误区**：端口映射只配到"内部端口 → 路径前缀"，用户要自己想办法从外面访问。

**业界实践**：E2B 和 CodeSandbox 都做**端口 → 自动公开 URL**：格式 `{sandbox_id}-{port}.domain`，沙箱内一起服务，外部立刻拿到预览链接。CodeSandbox 还支持 hot-reload。

**为什么**：Agent 越来越多地"在沙箱里起一个 Web 服务给用户看"（生成的前端、API、可视化）。如果用户要手动配 Ingress、配域名、等生效，这个"可预览 Agent"的场景就废了。自动公开 URL 把"起服务 → 拿到链接"压缩到零配置。

**避坑**：端口映射 UI 不要停在"内部端口 + 路由前缀"，要展示生成的**公开 URL** 并可一键打开。

## 洞察 7：`metadata` 业务标签是隐藏的刚需

**常见误区**：实例只有系统字段（ID/名称/状态），没有业务关联字段。

**业界实践**：E2B、Daytona、Modal 几乎所有平台都支持 `metadata`（任意 JSON），用来把沙箱关联到 `agent_id` / `session_id` / `tenant_id` / `task_id`。

**为什么**：控制台的实例列表会越来越长。没有业务标签，你无法回答"这 50 个沙箱分别属于哪个 agent、哪个会话、哪个租户"。`metadata` 是把**基础设施对象**和**业务语义**连起来的桥梁——列表过滤、成本归因、批量操作全靠它。

**避坑**：创建时强制（或强烈建议）填 tags + project + owner。列表页的过滤器要能按这些字段筛。这几乎零成本，但漏了后期补救极痛苦。

## 洞察 8：HPA 不能 scale-to-zero，KEDA 才能

**常见误区**：以为 K8s HPA 能做 scale-to-zero（缩到 0 副本省钱）。

**业界实践**：
- **HPA 原生不能 scale to 0**：`minReplicas` 默认 1，且从 0 状态无 metrics 源无法激活
- **KEDA 才能做 0↔1**：两阶段模型——Activation 阶段（KEDA 用 scaler 的 `IsActive` 决定 0↔1）+ Scaling 阶段（HPA 驱动 1↔N）
- **Knative** 也能 scale-to-zero（KPA），但有自己的两窗口参数

**为什么**：如果你打算用 K8s 做沙箱后端并想"闲时缩到 0 省钱"，直接上 HPA 会失败。必须引入 KEDA（或 Knative）做 0↔1 的激活层。这是个**架构选型级别的认知**，做错了要推倒重来。

**避坑**：scale-to-zero 的成本收益很诱人，但要算上**冷启动成本**（镜像拉取 + app init）。对延迟敏感的 Agent，考虑 `min-instances` 预热池或 Warm 快照来缓解唤醒延迟。

## 洞察 9：RBAC 要分"粗角色 + 细作用域"两层，exec/logs 是独立 subresource

**常见误区**：RBAC 只有"管理员/普通用户"两档，或者一个大而全的角色。

**业界实践**（8 个平台收敛）：
- **6 层角色模型**：Owner / Admin / Editor（邀请默认）/ Viewer / Billing / Custom
- **两层作用域**：粗（团队/组织）+ 细（项目/资源）
- **subresource 级授权**：`exec`（终端）、`logs`（日志）可独立授权——对标 K8s `pods/exec`、`pods/log`

**为什么**：终端（exec）是**最高权限的操作**——它能执行任意命令。你不应该让所有能"看"实例的人都能"exec"。把 exec/logs 拆成独立 subresource，才能实现"给运维 exec 权限、给开发只读 + 日志、给审计只读"的精细控制。

**避坑**：
- 邀请默认角色给 **Editor**（能创建修改，不能管成员/计费），不要给 Admin
- Custom 角色用 resource × verb 矩阵（对标 K8s Role rules 或 Northflank CRUD-per-resource）
- API token 要 scoped（对标 Fly macaroon），按 app/org 维度签发

## 洞察 10：成本治理是控制面职责，不能外包给用户

**常见误区**：提供资源，计费走另一套系统，控制台不管成本。

**业界实践**：成熟平台都把成本治理**内建到控制台**：
- 预算 cap + 阈值告警（50/80/100%）+ 超限动作（暂停新建 / 强制休眠）
- 配额（并发/CPU/内存/存储/GPU vs 上限）
- 按规格单价表 + 按秒计费
- 按项目/租户的成本归因

**为什么**：Agent 沙箱的失控成本风险极高——一个 agent 死循环跑 GPU 一晚上就是几百块。如果控制台不管成本，用户要么不敢用（怕烧钱），要么用了就出事。把预算/配额/告警做进控制台，是让用户**敢用**的前提。

**避坑**：成本治理不是 P2 锦上添花，是 **P0**。至少要有：配额上限（防单点失控）+ 预算告警（防累计失控）+ 按项目归因（防扯皮）。

---

## 一句话总结

> 一个成熟的 Agent 沙箱控制台，本质是在三组张力之间找平衡：**隔离强度 ↔ 启动延迟**、**控制粒度 ↔ 配置复杂度**、**状态保留 ↔ 存储成本**。上述 10 条洞察，每一条都是某个平台用真金白银（或线上事故）换来的平衡点——照着做能少走很多弯路。
